# 评分 Rubric 库

## code-quality-rubric

### 5 分 (卓越)
- 代码完全正确，所有测试通过
- 命名清晰、结构优雅、无冗余
- 适当的错误处理和边界检查
- 符合语言/框架最佳实践
- 可维护性极高

### 4 分 (优秀)
- 功能正确，主要测试通过
- 命名合理、结构良好
- 基本的错误处理
- 大部分符合最佳实践
- 可维护性好

### 3 分 (合格)
- 核心功能正确，部分边界条件未覆盖
- 命名基本合理，结构可接受
- 错误处理不完整
- 部分不符合最佳实践
- 可维护性一般

### 2 分 (不足)
- 功能部分正确，有明显 bug
- 命名混乱或结构差
- 缺少错误处理
- 明显违反最佳实践
- 难以维护

### 1 分 (不可接受)
- 功能基本不正确
- 代码混乱不可读
- 无错误处理
- 严重违反最佳实践
- 不可维护

## api-design-rubric

### 5 分
- RESTful 原则完全遵循（或 GraphQL schema 设计合理）
- 资源命名准确、一致
- 适当的 HTTP 方法和状态码
- 完整的错误响应格式
- 版本化策略清晰
- 分页/过滤/排序设计合理

### 3 分
- 基本 RESTful，有少量不一致
- 资源命名大体合理
- HTTP 方法正确但状态码不够精确
- 有错误响应但格式不统一
- 缺少版本化考虑

### 1 分
- RPC 风格伪装成 REST
- 命名混乱不一致
- HTTP 方法滥用
- 无标准错误格式
- 无版本化

## test-coverage-rubric

### 5 分
- 分支覆盖率 > 90%
- 覆盖所有公共接口
- 包含正向、反向、边界测试
- Mock 使用适当
- 测试命名清晰描述场景

### 3 分
- 分支覆盖率 60-90%
- 覆盖主要公共接口
- 主要正向测试，部分边界
- Mock 使用基本合理
- 测试命名基本可理解

### 1 分
- 分支覆盖率 < 60%
- 大量接口未测试
- 仅 happy path
- Mock 不当或缺失
- 测试命名不清

---

## doc-completeness-rubric

### 5 分
- 目的和范围明确
- 所有公共 API 有文档
- 包含使用示例
- 参数/返回值/异常完整描述
- 架构决策有说明

### 3 分
- 目的基本清楚
- 主要 API 有文档
- 少量示例
- 参数描述不完整
- 缺少架构说明

### 1 分
- 目的不明
- API 文档缺失严重
- 无示例
- 参数描述缺失
- 无架构说明

---

## 评分归一化

所有 rubric 评分从 1-5 归一化到 0-1：
```
normalized_score = (raw_score - 1) / 4
```

适应度计算使用归一化后的分数。
